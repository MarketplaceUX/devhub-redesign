Design Principles

{% if page.multipage %} {% include homePagination.html %} {% endif %}

General

  • Use sensible defaults. Always default to the most common or simplest use case.
  • Hide complexity from average users, but make it accessible for power users.
  • Use progressive disclosure to show the right content in the right context at the right time.
  • Use branching options to "funnel" the user from general choices to more specific ones, e.g., use "country selector" to minimize the number of payment providers to choose from.
  • Set expectations, and be transparent. Every page has "instructions" that tell the user what to do and what to expect to happen on that page.
  • Provide "help" content that helps developers make good/informed choices. Give brief "help" that summarizes the issue and point users to a deeper dive in MDN, if they want it.
  • Use clear/explicit language and tone - somewhere between the current language/tone and that used in our prototype.
  • Don't make me do it all in one sitting. If there is a flow, let user "save and exit" to return later.

Tactical

  • Support "testing" activities. Create features/guidance that help developers get feedback (e.g., private apps) before "public" launch and advice on fine-tuning.
  • Provide checklists. Wherever there is a list of stuff the developer must produce (or do), make a checklist available.
  • Don't re-invent the wheel on UI widgets. Developers aren't interested in devhub's UI. Don't make them "think" about it by asking them to use a widget that's new for the sake of being new.
  • Content creation/management of apps is a desktop experience. The vast majority of people prefer to fill out forms on the desktop where the keyboard is considerably larger and they have access to pointing devices. So we won't specifically design for a responsive experience.
