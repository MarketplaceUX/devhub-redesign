Design Story: Manage App Versions.

See the prototype

User flow - make a user flow diagram for this

This story covers:

  1. Managing manifest and packaged .zip files
  2. Managing versions (each version may have a new manifest or .zip file)
  3. Managing publishing status.

  1. Upload a new version of manifest. Add a release note if necessary.
  2. Submit it. The review team will test the updated version and view the changelog.
  3. When the app is approved, you can go to the App Details pages and modify the app’s screenshots, categories, details, etc. as needed. Remember: there’s only one detail page that’s shared for every version. But each version can have its own release note or packaged .zip file.
  4. Finally, when you’re ready, go back to the Manifest, Version & Publishing page, and click Publish. You can remain in the old version for as long as you like, and you can control when it’s published.

Changelog

  1. The functionality of the Manage Versions and App Status pages are now combined and located under Edit App. This new page is called Manifest, Version & Publishing (name TBD). It will no longer appear under the Actions menu.
  2. Consequently, the items under the Actions menu have become fewer. We have three: Edit App, User Reports and Analytics. If we don’t implement reports and analytics yet, we might even be able to dispense with the Actions menu entirely, which will simplify the navigation even more.
  3. To Submit a new app, you would do it on the My Submissions page. And to Add a new version, you would do it under the Manifest, Version & Publishing page. Makes sense, right? The function is separated cleanly.
  4. The process of adding a new version has been made simpler. Instead of taking you through the whole process from start to finish, we only have two fields that may change between one version and the next. These two fields are Manifest and Release Note.
